肾友笔记 v2.15.4 更新明细

## 功能新增
- wikiService 新增 fetchWikiFullPage：基于 action=parse API 获取维基百科完整词条（首段 lead + 章节段落 + 信息框 + 图集）
- WikiSection 类型：暴露章节标题 + 段落列表，详情页按章节结构化渲染
- EntityInfo 接口扩展：新增 lead / sections / infobox 字段，详情页全字段渲染词条描述、品类属性、形态参数、图集

## 逻辑修改
- 水果检索流程修复：去掉 searchWikiFruits 中附加的「 fruit 植物」后缀，改为只搜关键词本身（原后缀导致中英混合搜索命中率骤降）
- 药物检索逻辑修复：fetchBaikeInfo 不再持久化 null 到 localStorage，改为内存临时缓存（避免失败永久卡死，下次重启应用可重新联网尝试）
- 风控检测策略修复：isBlockedPage 由单关键词命中改为「强关键词单独命中 / 弱关键词对（百度安全验证 + 验证码）同时出现」二段式判定
- baikeService 失败缓存分离：成功结果落 localStorage（离线二次访问），失败结果仅存内存 Map
- doFetchFruit 改用 fetchWikiFullPage，summary API 降级为兜底

## UI 调整
- 水果详情页结构重排：图集（横向滚动）→ 词条简介（lead）→ 基本信息（infobox）→ 详细内容（sections 按章节）→ 每100g元素含量 → 食用建议
- 信息框键值对从 8 项扩展至 10 项，新增 sm:grid-cols-2 双列布局
- 图集横向滚动卡片：h-40 w-64，主图 + 多图合并去重展示
- 段落渲染：维基章节带 h5 小标题 + 段落列表，结构层次清晰

## 接口适配
- 维基百科 action=parse API（page={title}&prop=text|images|properties）适配完整词条解析
- 维基百科 search API 参数优化：srlimit 8→12，新增 srprop=snippet|sectiontitle，过滤 Help/Wikipedia/User 命名空间
- 百度百科 CORS 代理扩充：从 2 个增至 4 个（allorigins/corsproxy/thingproxy/codetabs 轮换）
- 百度百科 fetchHtml 超时：12s → 15s

## 数据策略优化
- 水果详情页图文：维基百科完整章节段落（替代前 200 字摘要），按 H2/H3 标题划分，过滤参考文献/外部链接/参见等元章节
- 药物详情页正文：段落数量从 8 段提升至 20 段，覆盖药理/适应症/不良反应/注意事项等全量字段
- 药物摘要：从前 200 字扩展至前 400 字，信息密度翻倍
- 药物信息框解析选择器扩展：新增 .basic-info / .J-basic-info / .basicInfo-item-wrap 等
- 维基百科图片解析：优先 .thumb/figure/.thumbimage，兜底 upload.wikimedia.org 全量 img

## 问题修复
- 问题1 修复：水果检索命中率异常——根因 searchWikiFruits 附加「 fruit 植物」后缀导致中英混合搜索命中率为 0，已移除
- 问题1 修复：水果详情页百科内容缺失——从 summary API（200 字摘要）升级为 parse API（完整章节段落+信息框+图集）
- 问题2 修复：药物详情图片加载失败——SmartImage 添加 referrerPolicy=no-referrer 解决 bkimg.cdn.bcebos.com 防盗链
- 问题2 修复：药物详情文本信息不全——正文段落数 8→20，摘要 200→400 字，多图 5→8 张
- 问题3 修复：药物检索误判为空——isBlockedPage 单关键词命中导致正常词条被风控误判，改为二段式判定
- 问题3 修复：null 持久化缓存导致永久失败——fetchBaikeInfo 失败仅存内存，重启应用后可重新尝试联网
- 维基百科消歧义页处理：尝试 name → name（水果）→ name（植物）→ name（食品）多个标题变体
